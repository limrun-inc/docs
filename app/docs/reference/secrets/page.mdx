# Organization secrets

Signing material outlives any single build: Google Play pins your upload key on first publish, Apple certificates get reused across profiles, an App Store Connect key is minted once and downloaded once. Organization secrets store that material on Limrun, typed and named, so builds resolve it by a stable identifier instead of passing files around.

A secret belongs to an organization and is addressed by a **type** and a **name**. The name is a natural key of the type, not a free-form label:

| Type | Name is the... | Written by |
|---|---|---|
| `androidSigningKey` | Android application ID | `lim gradle build --sign` / `--save-key` |
| `appleCertificate` | certificate serial number | the console's Apple signing flow |
| `appleProvisioningProfile` | provisioning profile identifier | the console's Apple signing flow |
| `appStoreConnectApiKey` | App Store Connect key ID | the API (no automated writer yet) |

Most users never call this API: the [Gradle signing flags](/docs/android/build-with-gradle#sign-a-release-aab) and the console's Apple flows manage their secrets automatically. The API matters when you migrate keys between organizations or environments, or audit what is stored.

## Semantics

**Writing is get-or-create.** A `PUT` to an existing type and name returns the stored secret untouched and discards what you submitted, so concurrent first writers converge on one key instead of racing. The response status tells you which happened: `201` means your material was stored, `200` means a secret already existed. Either way, **the response data is authoritative**: always use what comes back, never assume your submitted material was stored.

**Replace is explicit and destructive.** Setting `"replace": true` overwrites the stored data and the previous material is gone for good. For an `androidSigningKey` that destroys the upload keystore Google Play expects; treat replace as a last resort.

**Validation is type-aware.** Each type declares its required and optional data keys; unknown keys and empty required values are rejected with a `400`, so a typo cannot be silently stored and break every later build.

**Reads return the material.** Fetching a single secret returns its full data to the organization's own credentials. The list endpoint returns metadata only, newest first.

## Data keys by type

| Type | Required keys | Optional keys |
|---|---|---|
| `androidSigningKey` | `keystoreBase64`, `keystorePassword`, `keyAlias`, `keyPassword` | |
| `appleCertificate` | `certificateP12Base64`, `certificateType` | `certificatePassword`, `teamID`, `certificateID`, `serialNumber`, `expirationDate` |
| `appleProvisioningProfile` | `provisioningProfileBase64` | `certificateSerialNumbers`, `bundleIDs`, `deviceIDs`, `teamID`, `profileName`, `uuid`, `expirationDate` |
| `appStoreConnectApiKey` | `privateKeyP8Base64`, `keyId` | `issuerId`, `nickname`, `teamID`, `expirationDate` |

Binary material travels base64-encoded and must decode to non-empty bytes. Android keystore passwords and the key alias must contain only ISO-8859-1 characters, the encoding Gradle reads its properties with. The `appleProvisioningProfile` reference keys (`certificateSerialNumbers`, `bundleIDs`, `deviceIDs`) are comma-separated lists mirroring what the signed profile binds, so consumers can filter profiles without parsing every one; certificates are referenced by their Apple serial number.

## API

The secrets API is not in the SDKs; call it over REST with your API key. Names travel as path segments and must be URL-encoded.

Store (or fetch, if it already exists) an Android upload key:

```bash
curl -X PUT \
  "https://api.limrun.com/v1/organizations/$ORG_ID/secrets/androidSigningKey/com.example.app" \
  -H "Authorization: Bearer $LIM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "data": {
      "keystoreBase64": "'"$(base64 -i upload.jks)"'",
      "keystorePassword": "...",
      "keyAlias": "upload",
      "keyPassword": "..."
    }
  }'
```

Read one back, list, and delete:

```bash
curl "https://api.limrun.com/v1/organizations/$ORG_ID/secrets/androidSigningKey/com.example.app" \
  -H "Authorization: Bearer $LIM_API_KEY"

curl "https://api.limrun.com/v1/organizations/$ORG_ID/secrets?type=androidSigningKey&limit=100" \
  -H "Authorization: Bearer $LIM_API_KEY"

curl -X DELETE \
  "https://api.limrun.com/v1/organizations/$ORG_ID/secrets/androidSigningKey/com.example.app" \
  -H "Authorization: Bearer $LIM_API_KEY"
```

The list endpoint pages with `limit` (default 100, maximum 500) and `offset` (maximum 10000), and filters with `type`. Out-of-range paging values are rejected with a `400`, never silently clamped. Your organization ID comes from `GET /v1/whoami` with the same key.

## The console Secrets page

The **Secrets** page at [console.limrun.com](https://console.limrun.com) lists every stored secret with its type, name, and creation time, and deletes them. It also hosts the signing connections that write secrets: **Connect Play Console** for publishing and the Apple signing flow that stores certificates and provisioning profiles. Deleting a secret does not revoke anything with Apple or Google; it only removes Limrun's copy, and builds that relied on it will stop resolving it.
